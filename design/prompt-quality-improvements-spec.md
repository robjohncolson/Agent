# Prompt Quality Improvements Spec

Post-audit of the Codex agent prompts generated by `scripts/lib/build-codex-prompts.mjs`.
Compares the automated prompts against the user's natural workflow instructions to identify
pedagogical and structural gaps.

---

## Problem Statement

The prompt builder generates technically correct but pedagogically shallow prompts. The user's
natural instructions contain specific quality constraints that the builder omits entirely:

- **Worksheet**: No AP framework scoping — questions can drift beyond the video content
- **Blooket**: Optimizes for recall instead of conceptual discrimination; allows number-heavy
  questions and stylistically obvious correct answers
- **Drills**: No generator/grading-rules patterns embedded; no animation `.py` generation;
  hardcoded to unit 6 cartridge; no "create new cartridge" path

---

## 1. Embed AP Framework Scope in All Prompts

**Problem:** The framework file (`apstat_{unit}_framework.md`) is already read by
`extractTopicTitleFromFramework()` to extract the topic title, but the actual learning
objectives and essential knowledge are thrown away. Without framework context, the agent
can generate questions outside the AP-testable scope.

**Fix:** Add a new function `extractFrameworkSection(unit, lesson)` that:

1. Reads `apstat_{unit}_framework.md`
2. Extracts the full section for `TOPIC {unit}.{lesson}` — from the `## **TOPIC X.Y**` header
   through to the next `## **TOPIC` header (or end of file)
3. Returns the raw markdown string (typically 30-80 lines per topic)
4. Returns `null` if the framework file or section doesn't exist

Embed the result in all three prompts as a new section:

```
## AP Classroom Framework (Topic {unit}.{lesson})

This is the official AP scope for this lesson. Questions MUST align with these
learning objectives and essential knowledge statements. Do not test concepts
beyond what is listed here.

{frameworkSection}
```

**Placement:** After the video context section, before the requirements section — in all
three prompt builders.

**File:** `scripts/lib/build-codex-prompts.mjs`

## 2. Rewrite Blooket Prompt Requirements

**Problem:** The current Blooket requirements optimize for factual recall and reference
specific video examples. The user's actual pedagogy requires:
- Conceptual understanding over recall
- No mental math or calculation questions
- Answer choices that are stylistically indistinguishable
- Correct answers that require conceptual engagement to identify

**Fix:** Replace the `## Requirements` section in `buildBlooketPrompt()` with:

```
## Requirements

### Question Design
- Test CONCEPTUAL UNDERSTANDING, not recall of specific examples or numbers from the videos
- No questions requiring mental math or calculation — focus on reasoning and definitions
- Do not reference specific numerical examples from the videos (e.g., don't ask "in the
  hospital survey, what was the sample size?")
- Questions should test whether students understand WHY, not just WHAT
- Include questions that surface common misconceptions from the AP framework
- Target these categories:
  - Core definitions and when/why they matter (not just "what is X")
  - Procedural reasoning ("what must you check BEFORE doing X?", "what comes first?")
  - Misconception traps ("which of the following is a common error?")
  - Conceptual distinctions ("how does X differ from Y?")

### Answer Choice Design
- All 4 answer choices must be similar in LENGTH (within ~10 characters of each other)
- All 4 answer choices must be similar in COMPLEXITY and sentence structure
- The correct answer must NOT be identifiable by style alone — no "obviously longest"
  or "most detailed" correct answer
- Wrong answers must be plausible conceptual errors, not absurd or obviously wrong
- Randomize which position (1-4) holds the correct answer across questions

### Format
- 25-35 multiple choice questions covering all key concepts from {videoScope}
- Time limit: 20 seconds for all questions
- Do NOT include any extra text, headers, or explanations — just the raw CSV
```

**File:** `scripts/lib/build-codex-prompts.mjs` — `buildBlooketPrompt()` function

## 3. Add Worksheet Guardrails

**Problem:** The agent sometimes modifies CSS/JS infrastructure or generates questions
that don't match the fill-in-the-blank format well.

**Fix:** Add these constraints to the worksheet requirements:

```
### Constraints
- Do NOT add, remove, or modify any CSS rules or JS functions from the pattern file.
  Only change the HTML content sections.
- data-answer values must be concise (1-3 words). Include common misspellings and
  equivalent phrasings as pipe-separated alternatives.
  Good: data-answer="reject|reject H0|reject the null"
  Bad: data-answer="we reject the null hypothesis because the p-value is less than alpha"
- Use width 80px for single-word answers, 150px for short phrases, 250px for longer answers.
- Questions must follow the AP framework scope — do not test concepts beyond the
  learning objectives listed above.
```

**File:** `scripts/lib/build-codex-prompts.mjs` — `buildWorksheetPrompt()` function

## 4. Embed Generator and Grading-Rules Patterns in Drills Prompt

**Problem:** The drills prompt tells the agent to edit `generator.js` and `grading-rules.js`
but embeds zero examples of the existing code patterns. The prompt says "do not rely on
external repo exploration" but gives the agent no alternative. This results in the agent
either ignoring the instruction and reading files (slow, unreliable) or guessing patterns
(produces incompatible code).

**Fix:** Expand `buildManifestExcerpt()` (in `lesson-prep.mjs`) to also extract:

### 4a. Generator excerpt

Read the last generator function from `generator.js`. Strategy:
1. Read the full `generator.js` content
2. Find the last `case "l##-` block inside the `switch(modeId)` (or the last function
   block if it uses if/else)
3. Extract that block as the pattern example
4. If the file is too large, extract just the last 150 lines

Add to the manifest excerpt as `generatorExcerpt` and embed in the prompt:

```
## Generator pattern (last mode from generator.js)

Follow this exact pattern when adding new generator cases:

\`\`\`javascript
{generatorExcerpt}
\`\`\`
```

### 4b. Grading-rules excerpt

Same approach for `grading-rules.js`:
1. Extract the last `case "l##-` or if/else block
2. Embed as `gradingRulesExcerpt`

```
## Grading rules pattern (last mode from grading-rules.js)

Follow this exact pattern when adding new grading rule cases:

\`\`\`javascript
{gradingRulesExcerpt}
\`\`\`
```

### 4c. Reference the cartridge generation docs

Add to the drills prompt:

```
## Platform reference

The cartridge must comply with the Driller Platform cartridge spec. Key rules:
- generator.js must export generateProblem(modeId, context, mode)
- grading-rules.js must export gradeField(fieldId, answer, context) returning {score, feedback}
- Score values: 'E' (essentially correct), 'P' (partial), 'I' (incorrect)
- Mode IDs must match between manifest modes[] and progression.tiers[]
- Field IDs must match between mode.layout.inputs[].id and hints.perField keys
- Use shuffle bags for scenario variety; randomize numeric values and answer order
```

**Files:** `scripts/lesson-prep.mjs` (expand `buildManifestExcerpt`),
`scripts/lib/build-codex-prompts.mjs` (`buildDrillsPrompt`)

### Animation pattern

## 6. Parameterize Cartridge Path by Unit

**Problem:** `buildManifestExcerpt()` in `lesson-prep.mjs` hardcodes the cartridge path
to `cartridges/apstats-u6-inference-prop`. This breaks for Unit 7+.

**Fix:** Replace the hardcoded path with auto-detection:

```js
function findCartridgePath(unit) {
  const cartridgesDir = path.join(WORKING_DIRS.driller, "cartridges");
  const entries = readdirSync(cartridgesDir);

  // Look for apstats-u{unit}-* directory
  const match = entries.find(e =>
    e.startsWith(`apstats-u${unit}`) && statSync(path.join(cartridgesDir, e)).isDirectory()
  );

  if (match) {
    return match; // e.g. "apstats-u6-inference-prop"
  }

  return null; // No existing cartridge for this unit
}
```

Update `buildManifestExcerpt()` to accept unit as a parameter and use `findCartridgePath()`.

**File:** `scripts/lesson-prep.mjs`

## 7. Add New-Cartridge Creation Path

**Problem:** When no cartridge exists for the unit (e.g., starting Unit 7), the current
`buildDrillsPrompt()` assumes an existing cartridge to extend. The user's workflow has
a distinct "create cartridge" instruction for new units.

**Fix:** Add a `buildNewCartridgePrompt()` function to `build-codex-prompts.mjs`:

### 7a. Detection

In `step2_contentGeneration()`, after calling `findCartridgePath(unit)`:
- If a cartridge exists → use `buildDrillsPrompt()` (extend)
- If no cartridge exists → use `buildNewCartridgePrompt()` (create)

### 7b. New cartridge prompt

```js
export function buildNewCartridgePrompt(unit, lesson, videoContext, templateExcerpt) {
  // ...
}
```

The prompt should:

1. Reference the `_template/` cartridge directory as the structural pattern
2. Instruct the agent to create a new directory: `cartridges/apstats-u{unit}-{slug}/`
3. Generate all required files: `manifest.json`, `generator.js`, `grading-rules.js`,
   `ai-grader-prompt.txt`
4. Include the CARTRIDGE-GENERATION-PROMPT.md key rules inline (not the full 580 lines —
   extract the essential rules: manifest structure, generator export signature,
   grading-rules export signature, E/P/I scoring, mode ID conventions)
5. Include framework section and video context
6. Add the cartridge to `registry.json`
7. Generate animation `.py` files for each mode (same requirements as item #5)
8. Scope questions to "only content addressed in the video, not the full unit"

### 7c. Template excerpt

Add a function `buildTemplateExcerpt()` that reads from the `_template/` cartridge:
- `_template/manifest.json` structure
- `_template/generator.js` export pattern
- `_template/grading-rules.js` export pattern

Embed these in the new cartridge prompt.

**Files:** `scripts/lib/build-codex-prompts.mjs` (new function),
`scripts/lesson-prep.mjs` (branching logic in step2)
